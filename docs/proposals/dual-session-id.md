# Proposal: Dual Session ID Model

**Author:** Jeffrey Carlson  
**Date:** 2026-04-03  
**Status:** APPROVED — `placementSessionId` selected
**Decision Date:** 2026-04-03  
**Reviewers:** software-architect, product-manager

---

## Problem

The current SHARC spec (and SIMID) delegates session ID generation entirely to the creative. This works for single-ad scenarios but creates a gap when a container manages multiple simultaneous creatives:

- A page running leaderboard + interstitial + sticky footer simultaneously has no clean container-level session namespace
- Session IDs generated by different creatives could theoretically collide
- Container-side logging/analytics has no reliable identifier that it controls
- SIMID has the same limitation — likely never considered for multi-ad scenarios

## Proposal: Two Session IDs

Introduce an optional `containerSessionId` alongside the existing creative-generated `sessionId`.

### Bootstrap message (container → creative)
```json
{
  "type": "SHARC:bootstrap",
  "containerSessionId": "ctr-789-abc",
  "port": "[MessagePort]"
}
```

### createSession (creative → container)
```json
{
  "sessionId": "creative-generated-uuid",
  "containerSessionId": "ctr-789-abc"
}
```

## Design Properties

| Property | sessionId | containerSessionId |
|----------|-----------|-------------------|
| Generated by | Creative | Container |
| Purpose | Creative readiness signal + message correlation | Container-level session namespace |
| SIMID compatibility | ✅ Unchanged | ✅ Optional — old creatives ignore it |
| Breaking change | None | None |
| Required | Yes (existing) | No (new, optional) |

## Benefits

1. **Multi-ad management** — container assigns `containerSessionId` per ad slot before creative loads; clean namespace regardless of creative behavior
2. **Logging/analytics** — container can track sessions by its own ID without depending on creative-generated values
3. **Collision prevention** — even if two creatives generate the same UUID (astronomically unlikely but possible), the container can disambiguate by `containerSessionId`
4. **SIMID compatibility** — existing SIMID creatives and players are unaffected; this is a pure additive change
5. **SIMID v2 candidate** — could be proposed as a non-breaking addition to SIMID spec

## Questions for Review

1. Should `containerSessionId` be required in the bootstrap, or truly optional?
2. If creative doesn't echo `containerSessionId` back in `createSession` — what does the container do? Accept and proceed? Reject?
3. Should `containerSessionId` appear in ALL subsequent messages, or only in `createSession`?
4. Is this worth proposing to the SIMID working group now, or after SHARC ships?
5. Any naming concerns — `containerSessionId` vs `slotId` vs `adUnitId`?

## Implementation Impact

- Bootstrap message: add optional `containerSessionId` field
- `createSession` handler: read and store echoed `containerSessionId`
- All container-side logging: use `containerSessionId` as primary key
- Creative SDK: transparently echo `containerSessionId` from bootstrap — zero API surface change for creative developers
- Backward compatible: creatives that don't know about `containerSessionId` simply don't echo it
