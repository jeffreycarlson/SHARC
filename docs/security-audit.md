# SHARC Reference Implementation — Security Audit

**Version:** 0.1.0  
**Auditor:** Security Engineering Review  
**Audit Date:** 2026-04-03  
**Files Reviewed:**
- `src/sharc-protocol.js`
- `src/sharc-container.js`
- `src/sharc-creative.js`
- `src/test/index.html`

**Reference Documents:**
- `docs/architecture-design.md`
- `samples/SHARC/README.md` (IAB Tech Lab SHARC spec)

---

## Executive Summary

The SHARC reference implementation has a solid structural foundation. The MessageChannel transport is the right choice and provides meaningful isolation vs. raw `postMessage`. The sandboxed cross-origin iframe model is sound. However, **several implementation gaps exist that will propagate into every publisher deployment worldwide if left unaddressed before this becomes the normative reference**.

The most critical issues are:

1. The fallback postMessage transport (`*` targetOrigin on data messages) leaks the full session data to any listener on the page.
2. URL validation is entirely absent — a malicious creative can inject arbitrary URLs into the navigation and tracker flows.
3. The `allow-same-origin` sandbox attribute combined with a same-origin creative defeats the entire sandbox model.
4. The extension `requestFeature` method allows creatives to construct arbitrary message types, bypassing type validation.

None of these are show-stoppers, but all need to be fixed before v1.0 ships.

| Severity | Meaning |

| **Medium** | Exploitable in degraded or edge-case conditions. Defense-in-depth failures. Fix before wide publisher rollout. |

### SEC-001 — `allow-same-origin` sandbox attribute defeats cross-origin isolation

**Severity: Critical**

**Location:** `sharc-container.js`, `_createIframe()`

```javascript
iframe.setAttribute('sandbox', [
  'allow-scripts',
  'allow-same-origin',    // ← this one
  'allow-forms',
  'allow-popups',
  'allow-popups-to-escape-sandbox',
].join(' '));
```

**The problem:**

`allow-same-origin` combined with `allow-scripts` is documented by the HTML spec as explicitly defeating the sandbox:

> _"Setting both the allow-scripts and allow-same-origin keywords together when the embedded page has the same origin as the page containing the iframe allows the embedded page to simply remove the sandbox attribute and then reload itself, effectively breaking out of the sandbox."_

If the creative URL is served from the same origin as the publisher page (e.g., a CDN that serves both), a malicious creative can:

1. Access `document.referrer` or walk up the frame tree via `window.parent`
2. Remove its own sandbox attribute: `window.frameElement.removeAttribute('sandbox')`
3. Reload with full publisher-page access

This is a complete sandbox escape. Publisher DOM, cookies, localStorage, and all publisher-origin data are exposed.

**Rationale for including it here (the "MessageChannel needs it" argument):**

`allow-same-origin` is sometimes included to allow MessageChannel to function. This is incorrect — MessageChannel works fine across origins. The port is transferred; origin doesn't matter for MessageChannel communication after the transfer.

**Fix:**

Remove `allow-same-origin` entirely. Test that MessageChannel bootstrap still functions (it will). If the creative legitimately requires same-origin access for a specific feature, that access must be mediated through the SHARC protocol, not by relaxing the sandbox.

```javascript
iframe.setAttribute('sandbox', [
  'allow-scripts',
  // allow-same-origin REMOVED
  'allow-forms',
  'allow-popups',
  'allow-popups-to-escape-sandbox',
].join(' '));
```

### SEC-002 — Fallback postMessage uses `*` targetOrigin with full message payloads

**Severity: High**

**Location:** `sharc-protocol.js`, `SHARCContainerProtocol._sendMessage()` and `SHARCCreativeProtocol._sendMessage()`

```javascript
// Container fallback path:
this._fallbackTarget.postMessage(JSON.stringify(message), '*');

// Creative fallback path:
window.parent.postMessage(JSON.stringify(message), '*');
```

**The problem:**

The architecture design explicitly justifies `*` targetOrigin for the bootstrap message only, because that message carries only a port (no data). The fallback transport is different — it sends **full SHARC messages with session IDs, args payloads, and tracking URIs** to `*`. Every listener on the publisher page (third-party analytics, ad verification scripts, other ad iframes) can read these messages.

This affects:
- Session IDs (can be harvested for session hijacking in future attacks)
- `reportInteraction` tracker URIs (leak ad campaign data)
- `requestNavigation` URLs (leak user intent/clickthrough destinations)
- `environmentData` (may contain placement metadata and user context)

Additionally, the container's fallback transport checks `event.source !== creativeWindow` but doesn't validate origin. Any frame that has a reference to the container's window can impersonate the creative.

The fallback is described as "effectively zero real-world cases" in the architecture doc. If that's true, the fallback can be hardened or removed without practical impact.

**Fix (Option A — Harden the fallback):**

Use a known targetOrigin when the container creates the iframe. The container controls the `src` attribute; it knows the creative's origin at iframe creation time. Store the origin and use it in fallback sends:

```javascript
// At iframe creation time:
this._creativeOrigin = new URL(this.creativeUrl).origin;

// In fallback sends:
this._fallbackTarget.postMessage(JSON.stringify(message), this._creativeOrigin);
```

On the creative side, validate the sender origin against the expected container origin (passed as a parameter or derived from `document.referrer`).

**Fix (Option B — Remove the fallback):**

MessageChannel has 97%+ support per the architecture doc. Document that SHARC requires MessageChannel and remove the fallback entirely. This eliminates the attack surface.

### SEC-003 — No URL validation on `requestNavigation` — open redirect / arbitrary navigation

**Severity: High**

**Location:** `sharc-container.js`, `_handleRequestNavigation()`

```javascript
_handleRequestNavigation(msg) {
  const navArgs = msg.args || {};
  if (this._onNavigation) {
    this._onNavigation(navArgs);
  } else {
    // Default behavior: open clickthrough in new tab
    const { url, target } = navArgs;
    if (url && (target === 'clickthrough' || !target)) {
      try { window.open(url, '_blank', 'noopener,noreferrer'); } catch (e) { /* ignore */ }
    }
  }
}
```

**The problem:**

There is zero validation on the `url` field before it's passed to `window.open`. A malicious creative can pass:

- `javascript:alert(document.cookie)` — JavaScript execution in the publisher page context (depending on browser version and `noopener` handling)
- `data:text/html,<script>...</script>` — loads an attacker-controlled page
- `file:///etc/passwd` — on desktop browsers in certain configurations
- Deep links like `itms-services://` that trigger OS-level actions (store installs, app launches) without user intent

The `noopener,noreferrer` option is good and limits some of these risks, but it doesn't eliminate them. `javascript:` URIs in `window.open` behave inconsistently across browsers and have had a history of bypasses.

Even with the `onNavigation` callback provided, the spec says the container should validate parameters "before acting" (architecture §5.3) — but the default path does no validation, and there's no guidance to custom callback implementers.

**Fix:**

Add URL validation before any navigation occurs. Enforce an allowlist of safe schemes:

```javascript
function isNavigationUrlSafe(url) {
  try {
    const parsed = new URL(url);
    // Only allow https (and http for legacy), standard deep link schemes
    const safeSchemes = new Set(['https:', 'http:']);
    return safeSchemes.has(parsed.protocol);
  } catch {
    return false; // Not a valid URL
  }
}
```

For deep links and custom schemes (`deeplink`, `store`, `custom` targets), validate those scheme values against an allowlist configured by the publisher, not by the creative.

### SEC-004 — No URL validation on `reportInteraction` tracker URIs

**Severity: High**

**Location:** `sharc-container.js`, `_fireTrackers()`

```javascript
_fireTrackers(uris) {
  if (!uris || uris.length === 0) return Promise.resolve([]);
  const fireOne = (uri) => {
    return new Promise((resolve) => {
      // ...
      fetch(uri, fetchOptions) // ← uri is unchecked
```

**The problem:**

The container fires arbitrary URIs supplied by the creative via HTTP GET with no validation. A malicious creative can:

1. **SSRF (Server-Side Request Forgery):** Pass internal network addresses like `http://169.254.169.254/latest/meta-data/` (AWS metadata endpoint), `http://10.0.0.1/admin/`, or `http://localhost:8080/`. In server-side rendering environments or Node.js-based renderers, this could expose internal infrastructure.

2. **Data exfiltration at scale:** Since the container fires these on behalf of the creative, the requests come from the publisher's domain (or the user's browser, but attributed to the ad). A malicious creative can exfiltrate any data it has access to by encoding it in a tracker URL.

3. **Array unbounded:** There's no limit on the number of tracker URIs. A creative can pass 10,000 URIs, each triggering a `fetch()` call.

4. **Scheme abuse:** `file://`, `javascript:`, and other non-HTTP URIs can be passed. `fetch()` with `no-cors` mode on `file://` URIs has implementation-specific behavior.

**Fix:**

```javascript
function isTrackerUriSafe(uri) {
  try {
    const parsed = new URL(uri);
    if (!['https:', 'http:'].includes(parsed.protocol)) return false;
    // Optional: block private/loopback ranges at the container level
    // (browser fetch in 'no-cors' mode won't reach localhost from cross-origin pages,
    //  but server-side containers must validate this explicitly)
    return true;
  } catch {
    return false;
  }
}

// In _handleReportInteraction:
const MAX_TRACKERS = 20;
const { trackingUris = [] } = (msg.args || {});
const safeUris = trackingUris
  .slice(0, MAX_TRACKERS)
  .filter(isTrackerUriSafe);
```

### SEC-005 — `requestFeature` allows creative to construct arbitrary message types

**Severity: High**

**Location:** `sharc-creative.js`, `SHARCCreativeSDK.requestFeature()`

```javascript
requestFeature(featureName, args = {}) {
  if (this._dead) return Promise.reject(new Error('SDK is dead'));
  const messageType = `SHARC:Creative:request${this._capitalize(featureName.split('.').pop() || featureName)}`;
  return this._proto._sendMessage(messageType, { featureName, args });
}

_capitalize(str) {
  if (!str) return '';
  return str.charAt(0).toUpperCase() + str.slice(1);
}
```

**The problem:**

The `featureName` is taken directly from creative input and used to construct the message type string. A creative can craft:

```javascript
SHARC.requestFeature('com.iabtechlab.sharc.Close'); 
// → generates: 'SHARC:Creative:requestClose'
// Which is a valid message type the container listens to!
```

Or:
```javascript
SHARC.requestFeature('com.evil.x.FatalError');
// → generates: 'SHARC:Creative:requestFatalError'
```

This allows the creative to generate messages with any type string, including existing protocol message types, bypassing the SDK's public API guards (`_dead` checks, parameter validation, etc.). The container's listener for these messages may behave differently than expected.

More broadly: the message type namespace is a security boundary. Creative-side code should not be able to construct arbitrary type strings that land in the container's dispatch table.

**Fix:**

Validate `featureName` against a strict format before constructing the message type. Feature names must conform to the `com.[domain].sharc.[name]` namespace format:

```javascript
const FEATURE_NAME_RE = /^com\.[a-z0-9.-]+\.sharc\.[a-z][a-z0-9-]*$/i;

requestFeature(featureName, args = {}) {
  if (!FEATURE_NAME_RE.test(featureName)) {
    return Promise.reject(new Error(`Invalid feature name: ${featureName}`));
  }
  // ...
}
```

Additionally, consider maintaining a registry of extension message types and only dispatching to known types on the container side.

### SEC-006 — Session ID generated by creative; container trusts it unconditionally

**Severity: Medium**

**Location:** `sharc-protocol.js`, `SHARCContainerProtocol.acceptSession()`

```javascript
acceptSession(createSessionMsg) {
  this.sessionId = createSessionMsg.sessionId;
  this._resolve(createSessionMsg, {});
}
```

**The problem:**

The creative generates the session ID and the container adopts it unconditionally. The architecture doc acknowledges this is the same pattern as SIMID, but it has two issues:

1. **Session ID collision:** If a malicious creative sends a session ID that happens to match an active session on another ad slot (or a predictable value like `"0"` or `"test"`), the container will accept it. There's no uniqueness enforcement at the container level.

2. **Late session injection:** The `_onPortMessage` handler only skips session validation for `CREATE_SESSION` messages. But there's a window between when the bootstrap port message is received and when `createSession` arrives. During this window, the port is live but `sessionId` is empty string `''`. A message with `sessionId: ''` (empty string) will fail the check `sessionId !== this.sessionId` only if `this.sessionId` has been set — but on fresh instantiation it's also `''`. If a message arrives before `createSession` with a matching `sessionId: ''`, it will pass session validation.

**Fix:**

The container should validate that the session ID from `createSession` is a well-formed UUID (or at minimum, a non-empty, non-trivial random string) before accepting it. Optionally, the container can generate the session ID itself and send it to the creative in the resolve, rather than trusting the creative's value:

```javascript
// Option: Container generates session ID and sends it in the createSession resolve
acceptSession(createSessionMsg) {
  const providedId = createSessionMsg.sessionId;
  // Validate format
  if (!isValidUUID(providedId)) {
    this._reject(createSessionMsg, ErrorCodes.INIT_SPEC_VIOLATION, 'Invalid session ID format');
    return;
  }
  this.sessionId = providedId;
  this._resolve(createSessionMsg, {});
}
```

For the empty-string race, initialize `sessionId` to `null` (not `''`) and update the validation check accordingly.

### SEC-007 — No rate limiting on incoming messages — resource exhaustion

**Severity: Medium**

**Location:** `sharc-protocol.js`, `_onPortMessage()` / `_dispatchToListeners()`

**The problem:**

A malicious creative can flood the container with messages at any rate. There is no throttling, rate limiting, or circuit-breaker on the incoming message handler. Effects:

- **CPU exhaustion:** Each message triggers listener callbacks, Promise allocations, and response sends.
- **Memory exhaustion:** Each unanswered request-requiring message adds an entry to `_pendingResponses`. A creative can send thousands of `CREATE_SESSION` or `GET_FEATURES` messages, filling this map.
- **UI freeze:** On the main thread, a flood of messages can freeze the publisher page, degrading user experience.

Error code 2205 ("creative overloading message channel") exists in the spec but is never triggered by this implementation.

**Fix:**

Implement a simple token-bucket or sliding-window rate limiter:

```javascript
// Example: max 50 messages per second, burst of 10
_onPortMessage(event) {
  if (!this._rateLimiter.allow()) {
    this._sendFatalError(ErrorCodes.OVERLOADING_CHANNEL, 'Message rate limit exceeded');
    return;
  }
  // ... existing handling
}
```

Also cap `_pendingResponses` at a reasonable size (e.g., 100 entries) and reject new requests when the cap is hit.

### SEC-008 — Bootstrap port message: no type re-validation after event.ports check

**Severity: Medium**

**Location:** `sharc-protocol.js`, `SHARCCreativeProtocol._onBootstrapMessage()`

```javascript
_onBootstrapMessage(event) {
  if (
    event.data &&
    typeof event.data === 'object' &&
    event.data.type === 'SHARC:port' &&
    event.ports &&
    event.ports[0]
  ) {
    window.removeEventListener('message', this._bootstrapHandler, false);
    this._portReceived = true;
    this._usingMessageChannel = true;
    this._attachPort(event.ports[0]);
  }
}
```

**The problem:**

The check is `event.data.type === 'SHARC:port'` but there is no validation of `event.origin`. Any frame that can `postMessage` to the creative window — including other ad iframes on the same page — can send a fake bootstrap message with a compromised port. The creative will adopt the attacker's port, discarding the legitimate container's port.

This could be used to:
- Intercept all subsequent creative-to-container messages (man-in-the-middle)
- Send crafted container-to-creative messages (fake `Container:init` with malicious `environmentData`)

The architecture doc acknowledges the `*` targetOrigin is intentional for the container→creative direction. But on the creative side, there is no symmetric protection — the creative should know who its parent is.

**Fix:**

The creative should validate that the bootstrap message comes from its expected parent:

```javascript
_onBootstrapMessage(event) {
  // Only accept bootstrap from direct parent
  if (event.source !== window.parent) return;
  
  if (
    event.data &&
    typeof event.data === 'object' &&
    event.data.type === 'SHARC:port' &&
    event.ports &&
    event.ports[0]
  ) {
    // ...
  }
}
```

This is the symmetric check to the container's `event.source !== creativeWindow` check in the fallback transport. Add it here.

### SEC-009 — Creative log messages forwarded to `console.log` without sanitization

**Severity: Medium**

**Location:** `sharc-container.js`, `_registerProtocolListeners()`

```javascript
proto.addListener(CreativeMessages.LOG, (msg) => {
  this._onMessage && this._onMessage('received', msg);
  console.log('[SHARC Creative Log]', msg.args && msg.args.message);
});
```

**The problem:**

The `msg.args.message` value is logged directly to `console.log`. On the surface this seems harmless, but:

1. **Log injection:** If publisher tooling processes `console.log` output (log aggregation, security monitoring), a creative can inject fake log entries by embedding newlines or structured log format strings in the message.

2. **Prototype pollution via structured clone:** While Structured Clone prevents `__proto__` pollution in the message itself, the `message` field is still an arbitrary string that may contain characters interpreted by log processing tools (JSON parsers, SIEM regex rules).

3. **`msg.args` is unvalidated:** If `msg.args.message` is not a string (e.g., an object), `console.log` will serialize it as `[object Object]` which is benign, but a creative could pass a very large string to spam the developer console, or a string with formatting codes on certain platforms.

**Fix:**

Validate and truncate the log message before forwarding:

```javascript
proto.addListener(CreativeMessages.LOG, (msg) => {
  const rawMessage = msg.args && msg.args.message;
  if (typeof rawMessage === 'string') {
    const sanitized = rawMessage.substring(0, 1024); // cap at 1KB
    console.log('[SHARC Creative Log]', sanitized);
  }
});
```

Same fix applies to `Container:log` forwarding in `sharc-creative.js`.

### SEC-010 — `allow-popups-to-escape-sandbox` grants broad popup capability

**Severity: Medium**

**Location:** `sharc-container.js`, `_createIframe()`

```javascript
iframe.setAttribute('sandbox', [
  'allow-scripts',
  'allow-same-origin',
  'allow-forms',
  'allow-popups',
  'allow-popups-to-escape-sandbox',   // ← concern
].join(' '));
```

**The problem:**

`allow-popups-to-escape-sandbox` causes any popup opened by the sandboxed iframe to be **unsandboxed** — it inherits no sandbox restrictions. Combined with `allow-popups`, a creative can open a fully capable popup window with:

- Access to `window.opener` (pointing back to the creative iframe)
- Ability to redirect `window.opener.location` in some browser configurations
- Full scripting, storage, and network access from the creative's origin

This is a significant capability elevation. The spec allows creatives to request navigation via `requestNavigation`, which the container should handle. Creatives should not need direct popup access at all.

**Fix — Option A (Preferred):**

Remove both `allow-popups` and `allow-popups-to-escape-sandbox`. All navigation must go through `requestNavigation`. The container opens URLs on behalf of the creative.

**Fix — Option B (If popups are needed for some use case):**

Keep `allow-popups` but remove `allow-popups-to-escape-sandbox`. Sandboxed popups are more restrictive.

If popups are needed for social sharing widgets or similar, add explicit support via the extension framework so the container can control what popups are opened.

### SEC-011 — `_pendingResponses` map is unbounded

**Severity: Medium**

**Location:** `sharc-protocol.js`, `SHARCProtocolBase._sendMessage()`

```javascript
this._pendingResponses[messageId] = (responseData) => { ... };
this._port.postMessage(message);
```

**The problem:**

Every message sent that requires a response adds an entry to `_pendingResponses`, keyed by `messageId`. There is no:
- Maximum size cap
- Per-message TTL (only some messages have container-level timeouts)
- Cleanup of stale entries when the session becomes invalid

A malicious creative that receives `Container:init` but never resolves it, while also rapidly invoking any creative API that sends new messages (or a container that never resolves creative requests), can cause the map to grow indefinitely. On mobile WebView environments with limited heap, this leads to memory pressure and potential process kill.

Additionally: the `terminate()` method correctly rejects all pending callbacks, but `reset()` just sets `_pendingResponses = {}` — if `reset()` is called while promises are pending, those promises will never settle (memory leak / hanging Promises).

**Fix:**

Add a maximum size check before inserting:

```javascript
const MAX_PENDING = 100;
if (Object.keys(this._pendingResponses).length >= MAX_PENDING) {
  return Promise.reject(new Error('Too many pending responses'));
}
```

For `reset()`, reject all pending responses before clearing (same pattern as `terminate()`).

### SEC-012 — `requestPlacementChange` accepts arbitrary dimension strings → CSS injection

**Severity: Medium**

**Location:** `sharc-container.js`, `_applyIframeDimensions()`

```javascript
_applyIframeDimensions(dims) {
  if (!this._iframe) return;
  if (dims.width) this._iframe.style.width = typeof dims.width === 'number'
    ? `${dims.width}px`
    : dims.width;        // ← string passthrough
  if (dims.height) this._iframe.style.height = typeof dims.height === 'number'
    ? `${dims.height}px`
    : dims.height;       // ← string passthrough
}
```

**The problem:**

When `dims.width` or `dims.height` is a string (not a number), it is passed directly to `iframe.style.width/height`. A creative can pass:

```javascript
SHARC.requestPlacementChange({
  intent: 'resize',
  targetDimensions: {
    width: '100vw; position: fixed; top: 0; left: 0; z-index: 2147483647',
    height: '100vh'
  }
});
```

Browsers will parse this as a CSS style attribute value. While modern browsers limit inline style injection by stopping at the first invalid property, the `width` value itself can include units that push the iframe well outside its intended dimensions (`100vw`, `100vh`, negative values, etc.).

More critically, if the `containerEl` has CSS variables or the publisher page uses `contain` less strictly, large/negative dimension values can interfere with publisher page layout.

**Fix:**

Validate dimension values strictly — accept numbers only, or strings that are strictly `"{number}px"` or `"{number}%"`:

```javascript
function sanitizeDimension(val) {
  if (typeof val === 'number' && isFinite(val) && val >= 0) {
    return `${Math.round(val)}px`;
  }
  if (typeof val === 'string' && /^\d+(\.\d+)?(px|%)$/.test(val)) {
    return val;
  }
  return null; // reject invalid
}
```

Also enforce maximum dimension caps (e.g., no larger than the viewport) before applying.

**Severity: Low**

**Location:** `src/test/index.html`

```javascript
sharcContainer._protocol.sendFatalError(2200, 'Test fatal error from harness')
```

And the session-polling loop:
```javascript
const checkSession = setInterval(() => {

```

**The problem:**

The test harness directly accesses the internal `_protocol` object of `SHARCContainer`. This is a test harness, so it's not a production security issue — but if this pattern is copied by publishers integrating SHARC, they will have direct access to the raw protocol layer, bypassing the intentional API boundary.

### SEC-014 — `messageId` monotonicity not enforced on incoming messages

**Severity: Low**

**Location:** `sharc-protocol.js`, `_onPortMessage()`

**The problem:**

The architecture doc states: "The container validates: 1. The `sessionId` matches the active session. 2. The `messageId` is monotonically increasing (per sender)."

The session ID check is implemented. The `messageId` monotonicity check is **not implemented**. There's no tracking of the last-seen `messageId` from the counterpart, and no rejection of out-of-order or replayed messages.

This means a creative (or any code that intercepts the port) could replay a previously captured `requestClose` or `reportInteraction` message. In the context of MessageChannel (private port), replay requires compromising the port, which is hard — but in the fallback postMessage transport, a malicious third-party script that captured earlier messages can replay them.

**Fix:**

Track the last-seen `messageId` from the counterpart and reject messages that don't strictly increment:

```javascript
this._lastSeenRemoteMessageId = -1;

// In _onPortMessage, after session validation:
if (data.messageId <= this._lastSeenRemoteMessageId) {
  // Replay or out-of-order — reject
  return;
}
this._lastSeenRemoteMessageId = data.messageId;
```

### SEC-015 — No Content-Security-Policy set on the iframe

**Severity: Low**

**Location:** `sharc-container.js`, `_createIframe()`

**The problem:**

The iframe's `sandbox` attribute restricts capabilities, but the creative document's own Content-Security-Policy is the primary defense against XSS within the creative. The container sets no `csp` attribute on the iframe (available in modern browsers) and provides no guidance to publishers on what CSP the creative URL should return.

A malicious creative that contains XSS (or a legitimately-loaded creative that loads third-party scripts with vulnerabilities) has no second layer of defense from the container side.

**Fix (Partial — container side):**

The `<iframe>` element supports the `csp` attribute (experimental but widely supported) that applies a CSP to the embedded document:

```javascript
iframe.setAttribute('csp', "default-src 'self' https:; script-src 'self' https:; object-src 'none'");
```

Add guidance to the spec and reference implementation docs recommending that creative
